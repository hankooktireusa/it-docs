🧩 직접 권한 부여

이 문서는 사용자의 할당된 역할 외부에서 개별 권한(Permission)과 권한 수준(Privilege) 재정의를 직접 적용하는 방법을 설명합니다.

직접 부여는 가급적 드물게 사용해야 하며, 역할과 필터(예: No Pricing)로 충분하지 않은 경우에만 사용합니다.

직접 부여를 사용하는 경우

다음과 같은 경우 직접 부여를 사용합니다:

  • 사용자가 단 하나의 추가 권한 또는 권한 수준만 필요한 경우
  • 특정 상황에서 No Pricing 역할을 재정의해야 하는 경우
  • 레거시 그룹이 재사용 가능한 역할로 만들 가치가 없는 드문 조합을 가진 경우

직접 부여 작동 방식

직접 부여는 사용자별로 저장되며, 역할과는 별도로 존재합니다. 이를 통해 다음이 가능합니다:

  • 특정 권한에 대해 A, S, U, L 등 개별 권한 수준을 추가하거나 재정의
  • 역할 기반 권한을 보완하거나 재정의

이는 No Pricing과 같은 역할 및 필터 적용 후 평가됩니다.

예시

✅ 예시 1: 판매 보고서에만 단가(Unit Price) 추가

사용자 역할:

  • Report – Sales Report (A, S 부여)
  • No Pricing (U, L 제거)

→ 직접 부여 추가:

User: jdoe
Permission: Report → Sales Report
Privileges: U

이제 사용자는 다음과 같습니다:

  • 역할에서 A, S
  • 직접 부여에서 U
  • L은 여전히 No Pricing으로 제거됨

✅ 예시 2: 맞춤 송장 접근

사용자는 Invoice 역할이 필요 없으며, 재고 수준의 송장 데이터만 보고 싶어합니다.

→ 직접 부여 추가:

User: jsmith
Permission: Status → Invoice
Privileges: A, S

모범 사례

  • ✅ 드문 예외에만 직접 부여 사용
  • ❌ 직접 부여로 역할을 대체하지 말 것
  • 🔐 항상 직접 부여를 기록하고 정기적으로 검토
  • 🔄 여러 사용자가 동일한 재정의를 필요로 한다면, 역할을 업데이트하거나 작은 맞춤 역할 생성 권장
